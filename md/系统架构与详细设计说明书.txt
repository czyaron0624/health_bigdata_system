2. 系统建设目标

2.1 业务目标
本系统围绕卫生健康领域数据的采集、整理、分析和展示，建立一套可扩展的轻量级数据分析决策平台。主要目标如下:
1. 统一采集省级、国家级卫生健康新闻公告与统计信息
2. 从公告文本和上传材料中提取结构化卫生指标
3. 对人口、机构、人员、床位、服务、费用六大模块进行统计分析
4. 支持年度趋势分析、跨地区对比、趋势预测和异常预警
5. 通过管理端页面为业务人员提供可视化查看入口

2.2 技术目标
1. 以 MySQL 作为核心数据底座
2. 以 Flask 提供 Web 页面和 REST API
3. 以 Redis 提供轻量缓存与统计快照支持
4. 以 Spark 或 Python 批处理脚本完成分析任务
5. 保留向列式分析引擎和分布式文件存储演进的空间


3. 总体架构设计

3.1 架构概览
系统整体采用“采集层 + 存储层 + 处理层 + 服务层 + 展示层”的分层架构。

逻辑流向如下:
1. 采集层从国家卫健委、地方卫健委等来源抓取新闻公告
2. 原始数据进入 MySQL 原始新闻表
3. OCR/规则抽取程序将文本转换为结构化指标，写入指标事实表
4. Spark/Python 分析任务按业务主题生成分析结果表
5. Flask API 从原始表、视图或结果表读取数据，对外提供服务
6. 管理端页面调用 API 展示统计、预测和预警结果

3.2 分层架构

3.3 部署架构
当前项目采用单机开发部署方式，核心依赖如下:
1. MySQL 8.0 容器
2. Redis 6.2 容器
3. 本地 Python 运行 Flask 应用和数据脚本
4. 本地或单机 Spark 环境执行批处理脚本

部署关系:
1. docker-compose 提供 db 和 redis 容器
2. Flask 本地连接 localhost:3306 和 localhost:6379
3. Spark/Python 批处理脚本直接连接 MySQL
4. Web 管理端通过浏览器访问 Flask 服务 5000 端口


4. 技术架构设计

4.1 技术栈
1. Web框架: Flask 2.2.5
2. ORM/迁移: Flask-SQLAlchemy, Flask-Migrate, Alembic
3. 数据库驱动: PyMySQL, mysql-connector-python
4. 缓存: Redis
5. 数据处理: PySpark, Python 批处理脚本
6. 文档解析: pandas, PyPDF2, python-docx, openpyxl
7. 前端: Jinja2 模板 + 原生 HTML/CSS/JavaScript

4.2 关键配置
1. MySQL 连接串配置于 config.py
2. Redis 地址配置于 config.py
3. Flask 入口为 run.py 和 web_app/app.py
4. Docker 编排文件为 docker-compose.yml

4.3 当前技术实现特征
1. 事务型数据以 MySQL 为主
2. 分析结果仍回写 MySQL，未独立拆分数仓引擎
3. 管理端页面采用服务端模板，不是前后端完全分离模式
4. 部分分析任务由 Spark 执行，部分已改为纯 Python + MySQL 执行，目的是降低本地依赖和 JDBC 风险

当前系统明确使用以下地区口径:
1. 广西
2. 四川
3. 国家

设计规则:
1. 若原始指标表无 region 字段，则由 source_table 反推地区
2. guangxi_news 对应 广西
3. sichuan_news 对应 四川

这是系统中非常关键的口径约束，所有分析结果表和接口应使用统一口径。

6.3 时间口径设计
1. 原始指标表按 year, month 存储时间
2. ocr_metrics_yearly 采用 region + year 粒度
3. prediction_results 采用 predict_year 表示未来年份
4. anomaly_detection 使用 year 表示被检测年份

8. 核心流程详细设计

8.1 新闻采集流程
流程描述:
1. 爬虫程序访问目标网站
2. 解析列表页和详情页
3. 提取标题、正文、发布日期、链接、来源分类
4. 写入 MySQL 原始新闻表
5. 后续结构化任务按年份或批次继续处理

关键脚本:
1. crawlers/guangxi_health_crawler.py
2. crawlers/national_health_crawler.py
3. crawlers/sichuan_health_crawler.py

8.2 OCR结构化流程
流程描述:
1. 读取原始新闻正文或 OCR 文本
2. 根据规则匹配指标名和指标值
3. 提取 year, month, metric_key, metric_value
4. 写入 health_ocr_metrics
5. 通过 vw_metric_clean 过滤不合理值

关键脚本:
1. crawlers/backfill_ocr_metrics.py
2. crawlers/ocr_structurer.py

8.3 上传导入流程
流程描述:
1. 管理端上传 Excel/PDF/Word/TXT
2. document.py 将文件保存到 uploads
3. document_parser.py 依据扩展名解析内容
4. 根据字段特征识别目标表
5. 返回预览给前端
6. 管理员确认后写入 MySQL

设计特点:
1. 先预览再入库
2. 支持人口、机构、指标三类导入
3. 对 health_ocr_metrics 使用 upsert 防重复

8.4 六大分析流程
流程描述:
1. 读取 population_info, medical_institution, vw_metric_clean 等基础表
2. 按地区、年龄、级别、年度等维度聚合
3. 将结果写入 analysis_* 结果表
4. API 优先从结果表读取

关键脚本:
1. spark_job/six_modules_processor.py

8.5 年度汇总流程
流程描述:
1. 读取清洗后的指标表
2. 按 region + year 聚合关键指标
3. 写入 ocr_metrics_yearly

关键脚本:
1. spark_job/yearly_metrics_processor.py

8.6 区域对比流程
流程描述:
1. 读取 ocr_metrics_yearly
2. 读取 population_info 获取人口口径
3. 读取 institution_yearly_summary 或 medical_institution 获取机构口径
4. 计算每万人指标、机构数、服务效率指标
5. 按年度归一化形成评分
6. 写入 region_comparison

关键脚本:
1. spark_job/region_comparison_analyzer.py

8.7 趋势预测流程
流程描述:
1. 读取 ocr_metrics_yearly
2. 按地区和指标构造时间序列
3. 剔除明显的异常尾点和极端离群点
4. 使用线性回归预测未来 3 年
5. 计算置信区间和拟合优度
6. 写入 prediction_results

关键脚本:
1. spark_job/trend_prediction.py

8.8 异常预警流程
流程描述:
1. 读取 ocr_metrics_yearly 作为实际值
2. 读取 prediction_results 作为预测期望值
3. 若无预测重叠，则回退到近三年历史中位数
4. 计算偏离率
5. 按阈值生成告警级别
6. 写入 anomaly_detection

告警阈值:
1. 绝对偏离率 >= 20% 且 < 50% 为 warning
2. 绝对偏离率 >= 50% 为 critical

关键脚本:
1. spark_job/anomaly_detection.py

13. 性能设计

13.1 当前性能策略
1. 采用结果表降低页面实时聚合压力
2. 使用 Redis 保存部分统计快照
3. 使用 MySQL 索引提升按年份、指标键检索效率
4. 使用批处理离线生成对比、预测、预警结果

13.2 性能瓶颈点
1. 原始新闻表持续增长后，原始聚合查询会变慢
2. 文档导入目前为同步处理
3. Spark 在本地运行时受 JDBC 和内存配置影响较大
4. 管理端仍有部分旧接口实时读原始表

13.3 性能优化建议
1. 将管理端总览全面切换到结果表
2. 为高频接口增加统一缓存策略
3. 进一步将分析脚本转为纯 Python/MySQL 或统一调度
4. 引入定时任务框架统一执行批处理

16. 风险与改进建议

16.1 数据风险
1. 上游表格口径不统一会直接影响对比和预测结果
2. 未完全年数据会导致趋势预测和异常预警出现偏差
3. 历史机构表缺少明细名称时，只能做年度汇总分析

16.2 系统风险
1. 单机部署不适合生产级高并发
2. 任务执行尚未纳入统一调度和监控
3. 错误恢复与补数流程依赖人工

16.3 优化建议
1. 建立统一任务调度器，按日或按周生成结果表
2. 为关键表补充数据字典和质量校验规则
3. 增加任务执行日志表和运行状态页
4. 将人口和机构基础数据逐步标准化
5. 清理旧接口，统一所有页面读取结果表
6. 增加自动化测试与接口验收脚本


17. 结论

本项目当前已经形成一套完整的卫生健康数据分析系统雏形，其核心特点是:
1. 以 MySQL 为中心的数据底座
2. 以 Flask 为中心的页面与接口服务
3. 以 OCR 结构化指标为核心分析输入
4. 以结果表驱动的统计分析、预测分析和异常预警
5. 以管理端页面作为业务使用入口

从工程角度看，系统已经具备“可演示、可迭代、可继续工程化”的基础。
从下一阶段建设看，建议重点推进三件事:
1. 统一基础数据口径
2. 统一前端和 API 的结果表读取路径
3. 建立任务调度、日志、监控和测试体系
